     Wildcard System Implementation Plan

     Current State Analysis

     Existing Dice System Architecture

     1. DiceUIManager.cs - Handles all dice UI interactions
       - Uses standard Unity UI Button components with onClick listeners
       - Manages 6 dice buttons (3 shape + 3 building)
       - Implements water die panel with 5 building type buttons
       - No long-press detection currently implemented
     2. DiceManager.cs & DicePool.cs - Core dice logic
       - Fully implemented dice selection, rolling, auto-reroll for triples
       - Water die handling with building type selection
       - Star awarding for double rolls
     3. GameManager.cs - Game state
       - Has wildcardsUsed field but no implementation
       - No wildcard cost tracking or UI
     4. Input System - Unity's new Input System
       - Currently used for shape manipulation (movement, rotation, 
     placement)
       - Not used for UI/dice interactions (UI uses legacy Button system)

     Water Die Panel Reference Implementation

     - Trigger: Click on water die building face
     - UI: Modal panel with 5 building type buttons
     - Flow: Click ‚Üí Show panel ‚Üí Select building type ‚Üí Hide panel ‚Üí Apply 
     selection
     - Storage: waterDieChosenBuildingType in DiceManager

     Proposed Long-Press Wildcard Implementation

     Option 1: Long-Press on Dice Faces (User's Preference)

     Implementation Steps:
     1. Add Long-Press Detection to DiceUIManager
       - Use Unity's EventTrigger component with IPointerDownHandler and 
     IPointerUpHandler
       - Implement coroutine-based timing (e.g., 1-second hold)
       - Add visual feedback (button scaling, color change, timer ring)
     2. Create Wildcard Selection Panel
       - Two variants: Shape Type panel (6 options) and Building Type panel 
     (5 options)
       - Reuse water die panel pattern with appropriate buttons
       - Show correct panel based on which dice type was long-pressed
     3. Update Game Logic
       - Track wildcard usage count (max 3) in GameManager
       - Calculate escalating costs (-1, -2, -3 points)
       - Override selected die face with wildcard choice
       - Update validation to allow wildcard overrides
     4. UI Feedback & State
       - Add wildcard counter display
       - Show next wildcard cost
       - Visual indicator on dice eligible for wildcard
       - Disable wildcard when 3 used

     Pros:
     - Saves screen space (no extra UI button)
     - Intuitive interaction (long-press to access special options)
     - Consistent with mobile app patterns

     Cons:
     - More complex implementation (timing, visual feedback)
     - Discoverability issue (users might not find long-press feature)
     - Potential conflict with existing click handlers

     Option 2: Dedicated Wildcard Button

     Implementation Steps:
     1. Add Wildcard Button to UI
       - Simple button next to dice tray
       - Enabled/disabled based on wildcard count
       - Shows remaining uses and next cost
     2. Create Combined Selection Panel
       - Single panel with both shape and building type selection
       - Two-step or combined selection UI
     3. Same Game Logic Updates as Option 1

     Pros:
     - Simple implementation
     - Highly discoverable
     - Clear user feedback

     Cons:
     - Uses screen space
     - Less elegant than long-press

     Feasibility Assessment

     Technical Feasibility: MODERATE

     Challenges:
     1. No existing long-press infrastructure - Must implement from scratch
     2. Input system mismatch - UI uses legacy Buttons, not Input System
     3. Visual feedback complexity - Need timer, progress indicators
     4. State management - Handling concurrent click vs long-press

     Mitigations:
     1. Use EventTrigger components (Unity built-in)
     2. Implement simple coroutine timer
     3. Reuse water die panel patterns
     4. Leverage existing color coding system

     Implementation Complexity Breakdown

     Easy Components:
     - Wildcard selection panel UI (similar to water die panel)
     - GameManager wildcard tracking
     - Cost calculation logic

     Moderate Components:
     - Long-press detection with visual feedback
     - Panel type switching (shape vs building)
     - Integration with existing dice selection logic

     Complex Components:
     - Preventing conflicts between click and long-press
     - Handling state during long-press (what happens if user drags finger?)
     - Visual polish (smooth animations, feedback)

     Potential Problems & Edge Cases

     1. Input Conflicts

     - Problem: Click vs long-press ambiguity
     - Solution: Clear timing threshold (e.g., 1s), cancel on drag

     2. State Management

     - Problem: What if user long-presses during water die selection?
     - Solution: Disable wildcard when water die panel active

     3. Visual Feedback

     - Problem: Users might not understand long-press action
     - Solution: Tutorial hint, visual timer, haptic feedback

     4. Game Rules Integration

     - Problem: Wildcard should affect star awarding for doubles
     - Solution: Check original roll faces for star eligibility, not 
     wildcard choices

     5. Multiplayer Sync

     - Problem: Wildcard usage needs to sync across players
     - Solution: Include in game state sync (future consideration)

     Final Implementation Plan (Based on User Choices)

     Approach: Button First, Long-Press Later

     User Selection: Start with dedicated wildcard button, add long-press as
      enhancement later.

     Design Decisions Finalized:

     1. Panel Design: Separate panels for shape/building (context-sensitive)
     2. UI Placement: Near dice tray (integrated with dice UI)
     3. Button Design: TWO wildcard buttons - one for shape dice, one for 
     building dice
     4. Eligibility: Wildcard currently selected die of appropriate type; if
      none selected, default to first die in that pool
     5. Long-press: 1-second hold duration with visual feedback
     6. Feedback: Timer ring with button scaling during hold
